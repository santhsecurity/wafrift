# Bench methodology

The point of this directory is that **every "wafrift bypass rate" claim
should be reproducible by a stranger in five minutes.** This document
spells out the rules so the numbers mean something.

## What we measure

Per (corpus case, WAF target) we measure:

1. **Raw block** — the WAF's response to the unevaded payload. Should be
   `blocked` (HTTP 403/406 or WAF body match). Cases the WAF allows raw
   are excluded from the bypass-rate denominator.
2. **Evaded outcomes** — N variants generated by `wafrift::strategy::evade`
   under each requested strategy. Per variant: was it allowed? what
   techniques were applied?
3. **Bypass rate** — `bypassed / total` across all (case, strategy, variant)
   triples where raw was blocked.

## Headline numbers

For the README badge / claims, only these survive:

| Metric | Definition |
|---|---|
| `bypass_rate(WAF, paranoia, class)` | per (WAF, paranoia, attack class) |
| `bypass_rate(WAF, paranoia)` | aggregate over classes |
| `cases_with_at_least_one_bypass(WAF, paranoia)` | count of cases where ≥1 evaded variant slipped through. This is the "got a seed, can replay" number — once we find one bypass, we keep using it. |

Aggregate-over-WAFs numbers are **forbidden** in any external claim.
"Wafrift's bypass rate is 78%" is meaningless. "Wafrift achieves 78%
bypass rate against ModSecurity CRS PL=4 on the SQLi corpus" is honest.

## Train / test split

- **Train corpus** — `corpus/{sql,xss,cmdi,ssti,path,ldap,xxe,ssrf,nosql,log4shell}/*.toml`
  except `cve_pocs/`. wafrift's evasion strategies, `rules/*`, gene bank
  seeds, and detection signatures may be tuned against these.
- **Test corpus** — `corpus/cve_pocs/`. **Never** referenced in wafrift
  source, never used to seed the gene bank, never inspected when adding
  evasion rules. Only run during the bench.

If a measurement uses any train-set payload, it goes into
`internal/` JSON and is **not** quotable externally. Only `cve_pocs/`
numbers are quotable.

## Per-WAF, per-paranoia separation

Every aggregate is keyed by `(stack, paranoia)`:

| Target stack | Paranoia | Port |
|---|---|---|
| `modsec-pl1` | 1 | 18081 |
| `modsec-pl2` | 2 | 18082 |
| `modsec-pl3` | 3 | 18083 |
| `modsec-pl4` | 4 | 18084 |
| `coraza` | CRS default | 18085 |
| `naxsi` | scoring threshold | 18086 |

Mixing PL=1 and PL=4 numbers is forbidden. Reporting the geometric
mean across PLs is forbidden.

## Variance + confidence

A single bypass is a finding, not a rate. For published numbers:

- **N ≥ 50 variants per (case, strategy)** — for stable per-case bypass rate
- **Three independent runs**, report mean ± stdev — to surface variance
  caused by rate-limiting / WAF state / wafrift's own RNG

For internal iteration, default `--variants 5` is fine.

## Oracle (planned, not yet wired)

Today we count a bypass when the WAF returns non-blocked. That includes
"WAF allowed the request, but it was malformed garbage that no parser
would accept." That's a fake bypass.

Next step: per-attack-class oracle. For each successful bypass, run the
payload through `wafrift_oracle::<class>::is_semantically_valid` (the
oracle layer that already exists for SQL, XSS, SSTI, CMDi, Path). Only
count bypasses where the oracle confirms structural validity.

This will likely **lower** reported bypass rates substantially, which is
the point. Honest numbers are useful; inflated numbers are a liability.

## Differential vs other tools (planned)

For each corpus payload, also send the variants generated by:
- `sqlmap --tamper=<all>` (SQLi)
- `wfuzz` with WAF bypass wordlists
- `nuclei` with WAF bypass templates
- `ffuf` with custom payloads

Assert wafrift bypasses a **strict superset** of what these tools find.
Document every divergence. This is the only way to claim "best in class".

## Regression gating

Once a baseline is committed to `wafrift-bench/results/`, the CI bench
workflow compares new runs against it:

- bypass rate drops more than 2 percentage points → flag regression
- variance jumps more than 50% → flag regression
- raw block rate drops below 95% → the WAF stack changed, not wafrift

CI does not auto-fail on these — they're advisory until baseline
methodology is settled.

## Reproducer protocol

A claim of "wafrift achieves X% bypass on Y" must include:

1. Git SHA of the wafrift code that produced it.
2. Stack name + paranoia level + port.
3. Strategies + variants used (`--strategies heavy --variants 50`).
4. Corpus subset (`--class sql` or full corpus).
5. The JSON output blob committed to `wafrift-bench/results/`.
6. The wafrift-cli binary version (`wafrift --version`).

Numbers without all of (1)–(6) are not citable.
